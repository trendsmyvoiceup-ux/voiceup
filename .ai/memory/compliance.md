## GDPR

### Vote data is anonymous by design
No personally identifiable information is stored with a vote at MVP. Session-based tracking (browser/session token) is used for abuse prevention only, not for voter identity. If this changes, a GDPR impact assessment is required before implementation.

### Data minimization is not optional
Collect only what is needed to produce the core function (a vote count per comparison per session). Any additional data collection — IP address logging, device fingerprinting, behavioral analytics — requires a clear purpose and explicit GDPR justification.

### Session tokens must be purgeable
Even anonymous session-based votes must be purgeable on request. Design the session token system so that clearing a token purges all associated vote records. This is required by GDPR Art. 17 even for data that appears anonymous.

### Storage limitation applies
Vote records should have a defined retention period. "Keep forever" is not a valid answer. Define retention before the database schema is finalized (TASK-0045).

## AI Act (EU)

### Battle content is AI-generated — disclosure is required
Any content generated by the AI pipeline (Battle Designer, Planner) that appears on public pages must be disclosed as AI-generated. This is both a trust model requirement and will be a legal requirement under the EU AI Act for AI-generated content at scale.

### The platform is not a high-risk AI system
Human Signal's AI pipeline generates entertainment/opinion content, not decisions affecting fundamental rights. It is not a high-risk AI system under Annex III of the EU AI Act. However, transparency and human oversight requirements still apply.

### Human oversight is built into the pipeline
The Reviewer gate (review.json.approved) and the Founder validation step in Creator Studio constitute the human oversight mechanism. Never automate this gate away without a Founder decision and a compliance review.

## Trust model principles

### "Not scientific polling" disclosure is mandatory everywhere
Any surface displaying vote counts or percentages — public battle page, API responses, Studio dashboard, any future consumer — must include the disclosure: these results are not scientifically rigorous. This is a core trust model requirement. It protects both users and the company.

### AI agents consuming platform data need the same disclosure
If a future AI agent reads vote data as input, it must receive the "not scientific polling" disclosure in its context. Platform signals cannot be laundered into authoritative-looking data via an agent intermediary.

### Trust is layered, not gating
The trust model explicitly rejects gatekeeping participation based on trust tier. Anyone can vote at MVP. Trust tiers (verified identity, weighted signals) are a future upgrade path that runs alongside open participation, not a replacement for it.

## Security conventions

### No authentication at MVP
The admin and studio routes have no authentication in V1. They are protected only by obscurity (direct URL required, not linked from public nav). This is acceptable for a single-founder, local-use tool. Authentication is required before any third party accesses internal routes.

### No sensitive data in localStorage
localStorage stores only UI state (Studio status, approval state, published timestamps). No vote data, no tokens, no PII is ever written to localStorage.

### Rate limiting is an MVP requirement, not a future feature
The trust model requires basic rate limiting on the vote endpoint at MVP. Unlimited voting with session token only is not sufficient. Rate limiting by IP (server-side) is the minimum bar before the voting API is exposed to real users.

## Accessibility

### Dark mode is the only mode currently
All design work assumes `<html class="dark">` is always applied. Light mode is not implemented. Do not make design decisions that would break in light mode — but do not invest in light mode until the product is validated.

### No accessibility audit yet
No formal accessibility review has been conducted. Before any public launch, WCAG 2.1 AA compliance is required. Prioritize: keyboard navigation for voting, sufficient color contrast (check oklch values against WCAG), focus indicators on interactive elements.
