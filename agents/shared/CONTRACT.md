# Content Factory — Pipeline Contract

This is a **permanent architectural decision** (founder decisions,
TASK-0025, extended by TASK-0031 to add the Reviewer gate, and by the
Content Validation sprint to add batch generation and reporting).
It replaces the previous strictly sequential workflow (TASK-0024).

## Authoritative architecture

```
Founder
  ↓
Planner                                  ← generates a BATCH: one proposal
  ↓  output/proposals/<slug>.json           per unique subject pair in the
  │  (one per pair in the category)         given category, not just one
  │
  ▼  for EACH battle in the batch:
Battle Designer
  ↓  output/battles/<slug>/         ← single source of truth (the "Battle Package")
Reviewer
  ↓  output/battles/<slug>/review.json
  │
  │  (gate: if review.json.approved === false, Website and Publisher do
  │   not run for THIS battle — the batch continues with the next one)
  │
  ├── Website          (reads the Battle Package independently)
  ├── Publisher         (reads the Battle Package independently, writes output/published/<slug>/)
  └── Future consumers  (Instagram, X, YouTube, API, MCP, etc. — read independently)
  │
  ▼  after every battle in the batch has been processed:
Report
  ↓  output/reports/<category>-<timestamp>.json
```

There is no other valid order. Any earlier draft describing a strictly
sequential Website → Publisher dependency (or any in-memory handoff) is
superseded and must not be followed. A rejected battle is a normal,
expected outcome of a batch run, not a pipeline failure — it is recorded
in the report and does not stop processing of the remaining battles.

## The Battle Package is the single source of truth

Once the Battle Designer writes `output/battles/<slug>/`, that folder is
authoritative. The Reviewer adds `review.json` to the same folder — it does
not become a second source of truth, it is metadata *about* the package.
Every downstream consumer — Website, Publisher, or any future consumer —
reads the Battle Package independently and **never** reads another
consumer's output or waits on another consumer to finish. There is no
"Website then Publisher" or "Publisher then Website" ordering: both (and
any future consumer) can run in any order, in parallel, or not at all,
without affecting each other. The only thing that gates them both equally
is the Reviewer's approval — that is a shared gate, not a dependency on
each other.

## Handoff artifacts — all on disk, no in-memory contracts

| Step | Producer | Artifact | Location |
|---|---|---|---|
| 1 | Planner | Battle proposal | `output/proposals/<slug>.json` |
| 2 | Battle Designer | Battle Package | `output/battles/<slug>/` |
| 3 | Reviewer | Review result (gate) | `output/battles/<slug>/review.json` |
| 4a | Website | Proposed code diff | `apps/web/src/lib/comparisons.ts` (uncommitted) |
| 4b | Publisher | Publication package | `output/published/<slug>/` (derived from the Battle Package, never the reverse) |

Steps 4a and 4b are independent siblings, not a sequence. Neither reads the
other's output, and neither requires the other to have run. Both require
step 3 (`review.json.approved === true`) to have happened first.

## Battle Package contents

`output/battles/<slug>/` always contains:

- `manifest.json` — package metadata (slug, schema version, source proposal, file list)
- `battle.json` — structured battle data (matches `Comparison`/`Subject` shape)
- `status.json` — independent per-consumer status tracking (each consumer writes only its own key)
- `script.txt`
- `caption.txt`
- `hashtags.txt`
- `image_prompt.txt`
- `video_prompt.txt`
- `review.json` — added by the Reviewer once review has run (see `agents/reviewer/OUTPUT_SPEC.md`)

See `output/battles/apple-vs-android/` for the reference template.

## Published package contents

`output/published/<slug>/`, generated by the Publisher only after Reviewer
approval, always contains:

- `README.md`
- `tiktok/` — `caption.txt`, `hashtags.txt`, `script.txt`, `video_prompt.txt`, `image_prompt.txt`
- `instagram/` — `caption.txt`, `hashtags.txt`
- `youtube/` — `title.txt`, `description.txt`, `hashtags.txt`

## Batch run report

After processing every battle in a category's batch, the runner writes one
report to `output/reports/<category>-<timestamp>.json`:

```json
{
  "category": "technology",
  "startedAt": "ISO timestamp",
  "finishedAt": "ISO timestamp",
  "executionTimeMs": 17,
  "totalBattles": 10,
  "approved": 10,
  "rejected": 0,
  "averageScore": 100,
  "generatedPackages": [
    { "slug": "apple-vs-android", "title": "Apple vs Android", "score": 100, "approved": true }
  ]
}
```

`/admin`'s Reports section reads the most recent file in `output/reports/`
to show last execution, approved, rejected, and average score.

## Runner (`scripts/run-pipeline.ts`) exit codes

- `0` — the batch ran to completion. This includes runs with one or more
  rejected battles — a rejection is an expected, recorded outcome, not a
  pipeline failure, and does not stop the batch.
- `1` — usage error, or the category itself could not be planned at all
  (e.g. unknown category — nothing to batch).

## Shared helper module

`scripts/fsutil.ts` holds the small filesystem/string helpers
(`readJSON`/`writeJSON`/`writeText`/`ensureDir`/`slugify`/`updateStatusKey`/
etc.) used by every script in `scripts/`, to avoid duplicating the same
read/write logic in each agent. This is internal code reuse within
`scripts/` only — it is not the same thing as the deferred "shared package"
between `scripts/` and `apps/web` (see `scripts/catalog.ts`'s header
comment), which remains a separate, still-open tradeoff.

## Output folders

- `output/proposals/` — Planner output (one file per pair in a category's batch).
- `output/battles/` — Battle Packages (the source of truth), each including a `review.json` once reviewed.
- `output/published/` — publication packages generated by the Publisher. No API calls, no uploads happen here — package generation only.
- `output/reports/` — one JSON file per batch run, written by the runner (see "Batch run report" above).

## Rules for every handoff

- Every handoff must exist on disk. No agent passes data to another agent
  in-memory or "directly" — each step reads its input from the file
  location documented above and nothing else.
- The Battle Designer reads only `output/proposals/<slug>.json`. It does not
  read anything from Website, Publisher, Reviewer, or any other consumer.
- The Reviewer reads only the Battle Package (`output/battles/<slug>/`),
  not the original proposal and not Website's/Publisher's output.
- Website and Publisher (and any future consumer) read only
  `output/battles/<slug>/`. They must not read from each other, depend on
  each other's status in `status.json`, or assume any execution order
  between them. Both run only after the Reviewer has approved the package —
  that gate comes from the Reviewer, not from one another.
- Each agent writes only to its documented output location — never edits
  files outside its `OUTPUT_SPEC.md` scope. A consumer writing its own
  `status.json` key does not count as depending on another consumer.
- No agent commits, pushes, or calls an external API. See `skills/git.md`.
- No agent invents content beyond what its `INPUT_SPEC.md` provides — if an
  input is missing or ambiguous, the agent stops and surfaces the gap rather
  than filling it in. The Publisher in particular must only reformat
  existing Battle Package content, never author new copy.
- Every agent (including future consumers) must be independently
  executable: given a valid Battle Package on disk, it runs without
  needing any other consumer to have run, in any session, process, or day.

## Versioning

This fan-out architecture (Battle Package as single source of truth,
Reviewer as a shared approval gate, Website/Publisher/future consumers all
independent of each other) is a permanent architectural decision and
supersedes any prior sequential ordering. Each agent's
`INPUT_SPEC.md`/`OUTPUT_SPEC.md` must stay in sync with this file; if they
diverge, this file is authoritative.
