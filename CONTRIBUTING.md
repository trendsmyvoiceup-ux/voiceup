# Contributing

## Principles

- Document decisions before implementing them (see `DECISIONS.md`).
- Keep changes small and reviewable.
- No application code or dependencies until the foundation docs are approved.

## Workflow

1. Open an issue using the appropriate template in `.github/ISSUE_TEMPLATE/`.
2. Create a branch named `type/short-description` (e.g. `docs/add-architecture-notes`).
3. Open a pull request using the PR template.
4. At least one review required before merge.

## Commit messages

Use concise, present-tense messages describing *why*, not just *what* (e.g. `Add database schema decision rationale`).

## Code style

To be defined once a tech stack is chosen (see `docs/03-ARCHITECTURE.md`).
